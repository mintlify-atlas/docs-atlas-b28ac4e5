## Reporting Bugs

### Before Submitting a Bug Report

<AccordionGroup>
  <Accordion title="Investigation Checklist">
    - Ensure you're using the latest version
    - Verify it's a bug, not an environment issue
    - Check if the bug is already reported
    - Search Stack Overflow and other resources
    - Collect detailed information about the bug
  </Accordion>
  
  <Accordion title="Information to Collect">
    - Stack trace or error messages
    - OS, platform, and version
    - Node.js, npm, and package versions
    - Your input and the actual output
    - Steps to reliably reproduce the issue
  </Accordion>
</AccordionGroup>

### How to Submit a Bug Report

<Warning>
  **Security Issues**: Never report security vulnerabilities publicly. Email sensitive bugs to souopsylvain@gmail.com
</Warning>

1. Open a [new issue](https://github.com/brayanj4y/rushed-ai/issues/new)
2. Use a clear, descriptive title
3. Explain expected vs. actual behavior
4. Provide detailed reproduction steps
5. Include all collected information

**What happens next:**

- The team will label your issue
- We'll attempt to reproduce it
- If reproducible, it will be marked `needs-fix`
- The issue is ready for implementation

### Setting Up Development Environment

<Steps>
  <Step title="Fork the Repository">
    Create your own fork of the repository on GitHub.
  </Step>
  
  <Step title="Clone Your Fork">
    ```bash
    git clone https://github.com/YOUR_USERNAME/rushed-ai.git
    cd rushed-ai
    ```
  </Step>
  
  <Step title="Install Dependencies">
    ```bash
    npm install --legacy-peer-deps
    ```
  </Step>
  
  <Step title="Set Up Environment Variables">
    Copy `.env.example` to `.env.local` and fill in required values:
    
    ```bash
    cp .env.example .env.local
    ```
    
    Required environment variables:
    
    ```env
    # Convex
    CONVEX_DEPLOYMENT=
    NEXT_PUBLIC_CONVEX_URL=
    NEXT_PUBLIC_CONVEX_SITE_URL=
    RUSHED_CONVEX_INTERNAL_KEY=
    CONVEX_DEPLOY_KEY=
    
    # Anthropic
    ANTHROPIC_API_KEY=
    
    # Clerk Authentication
    CLERK_JWT_ISSUER_DOMAIN=
    NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY=
    CLERK_SECRET_KEY=
    
    # Dodo Payments
    DODO_PAYMENTS_ENVIRONMENT=test_mode
    DODO_PAYMENTS_API_KEY=
    DODO_WEBHOOK_KEY=
    ```
  </Step>
  
  <Step title="Set Up Convex">
    ```bash
    npx convex dev
    ```
    
    This will set up your Convex backend and initialize the database schema.
  </Step>
  
  <Step title="Start Development Server">
    ```bash
    npm run dev
    ```
    
    The application will be running at `http://localhost:3000`.
  </Step>
</Steps>

### Component Guidelines

```typescript
// Use functional components with hooks
import { useState } from 'react';

interface MyComponentProps {
  title: string;
  count?: number;
}

export function MyComponent({ title, count = 0 }: MyComponentProps) {
  const [value, setValue] = useState(count);
  
  return (
    <div>
      <h1>{title}</h1>
      <p>Count: {value}</p>
    </div>
  );
}
```

**Component best practices:**

- Use functional components with hooks
- Keep components small and focused
- Use proper prop typing with TypeScript
- Follow file naming: `kebab-case.tsx`
- Place components in appropriate directories

## Git Workflow

### Branch Naming

```bash
# Feature branches
feature/user-authentication
feature/export-to-github

# Bug fix branches
fix/login-validation
fix/credit-calculation

# Documentation branches
docs/api-reference
docs/setup-guide
```

### Commit Messages

Follow these guidelines:

1. Use present tense: "Add feature" not "Added feature"
2. Use imperative mood: "Move cursor to..." not "Moves cursor to..."
3. Limit first line to 72 characters
4. Reference issues and PRs after the first line

**Optional commit emoji guide:**

- ✨ `:sparkles:` - New feature
- 🐛 `:bug:` - Bug fix
- 📚 `:books:` - Documentation
- ♻️ `:recycle:` - Refactoring
- 🎨 `:art:` - Code formatting/structure
- ⚡️ `:zap:` - Performance improvement
- 🔒 `:lock:` - Security

### Example Workflow

```bash
# Create a new branch
git checkout -b feature/new-export-option

# Make changes and commit
git add .
git commit -m "Add CSV export option for project files"

# Push to your fork
git push origin feature/new-export-option

# Open a pull request on GitHub
```

## Pull Request Process

<Steps>
  <Step title="Create a Feature Branch">
    Create a new branch for your changes from `main`.
  </Step>
  
  <Step title="Make Your Changes">
    Implement your feature or fix with clear, focused commits.
  </Step>
  
  <Step title="Test Thoroughly">
    Ensure all existing tests pass and add new tests for your changes.
  </Step>
  
  <Step title="Update Documentation">
    Update relevant documentation to reflect your changes.
  </Step>
  
  <Step title="Submit Pull Request">
    Open a PR with a clear description of your changes and link any related issues.
  </Step>
  
  <Step title="Respond to Feedback">
    Address any review comments and make requested changes.
  </Step>
</Steps>

<Note>
  Keep pull requests focused on a single change. If you have multiple unrelated improvements, submit separate PRs.
</Note>
